## Current Progress

- **Phase:** UI redesigned around the "Signal." dark dashboard prototype, with full admin CRUD over shiller profiles, a live **karma trend**, and now a **4-card KPI dashboard** driven by configurable **per-shiller weekly quotas** (comments + posts) with real **weekly post counts** read from Reddit. Data is comment-metrics + karma-Δ + weekly post counts + quota progress; payroll and post-history views remain deferred.
- **Last completed:** **Comment upvotes shown on the account-comments view + Newest / Most-upvoted sort (frontend-only).** The per-account comments page (`/reddit-accounts/:accountId/comments`) now renders each comment's upvote **score** in the meta row (green up-arrow when ≥ 0, matching Activity History) — the `score` was already delivered end-to-end (`toRedditComment` → `RedditComment.score`) but never rendered. Added a **Newest / Most upvoted** segmented-pill toggle (new global `.seg-pills` class copied from the topbar `.range-pills`; no new Material module) to **both** the comments page and Activity History. Sort is client-side over already-loaded comments (`sortedComments` computed: newest = source as-is, top = copied array sorted by score desc, tiebreak newest) — no network calls, never mutates the source signal array. History's "Most upvoted" ranks the recency-capped 40-item feed (documented as a `// Reason:` limitation). Verified: frontend `ng build` clean (400.01 kB, under budget). **Not browser-clicked this session** — verified at AOT-build level; sort logic is pure/inspected.
- **Previously completed:** **Dashboard caching freshness fix — comments no longer stay hidden after a refresh.** Fixed two faults that could hide a recent comment indefinitely and added a manual Refresh. (1) `AccountWeekMetric.refreshedAt` was `@updatedAt`, so the refresh *failure/no-op* path (which writes `update: {}` to preserve counts) still bumped the staleness clock — a repeatedly-failing account looked perpetually "fresh (<5 min)" and stopped being re-polled. `refreshedAt` is now **explicitly set** (only a successful re-poll marks a row fresh; cold-failure rows are stamped epoch 0 so they retry in the background; `update: {}` now preserves both counts *and* the old timestamp). Client-only change — **no migration**. (2) A browser refresh couldn't force fresh data. Added `?refresh=true` (→ `DashboardQueryDto.refresh` → `getDashboard(force)`) that re-polls Reddit synchronously for every in-scope account (bypasses the cache, returns `refreshing:undefined`), surfaced as a **Refresh button** on the dashboard (spins, keeps current numbers visible). The frontend's silent stale re-fetch was also made a bounded chain (≤4 × ~4 s) instead of a single 5 s one-shot. While investigating, the comment/post crawls were confirmed to rely on Reddit's **default newest-first ordering** — `sort=new` is **deliberately not** passed (it serves a stale sorted-listing edge cache that freezes counts at old values), now documented inline in `getComments`/`getWeeklyPostCount`. Verified: backend `nest build`+eslint + frontend `ng build` (399.61 kB, unchanged) clean; `prisma migrate status` = up to date (no drift); **live** (10 accounts) — normal load `refreshing:true` (~cache) vs `refresh=true` ~41 s synchronous Reddit re-poll returning `refreshing:undefined`, immediately-following load 282 ms fresh (`refreshing:undefined`, confirming the clock is set on success), `refresh=banana`→200 (coerced false). Force-refresh spinner/History not browser-clicked this session.
- **Earlier:** **Admin dashboard now scales by grouping accounts under their shiller.** The manager view no longer renders one chart bar + one table row per Reddit account (which turned into unreadable slivers past ~15–20 accounts). Both inline-SVG charts are now aggregated **per shiller** (bounded to the small shiller count) and the "Managed accounts" table is a **grouped, expandable** list — one summary row per shiller (account count · summed comments · summed karma · Δ this week · comment-quota % · N/M active), sorted by weekly comments desc, that drills down to that shiller's individual accounts. Backend is a tiny additive projection change (no migration): admin dashboard rows now also carry `ownerId`/`weeklyCommentQuota`/`weeklyPostQuota` (already-loaded values) so the frontend can build per-shiller rollups + quota % and link each group to the existing `/shillers/:id` page. Shiller (employee) view and the 4 fleet-KPI cards are unchanged.
- **Even earlier:** **Dashboard metrics cache (stale-while-revalidate).** The dashboard no longer calls Reddit in the request path — it serves per-account weekly counts from a new `AccountWeekMetric` table plus current karma/status from `RedditAccount` (new nullable `RedditAccount.karma`; migration `add_dashboard_metric_cache`). Reads are DB-only and instant; when a cached metric is older than 5 min (`METRICS_STALE_MS`) a **background** refresh re-polls Reddit and upserts the cache without blocking the response (payload gained a `refreshing?` flag; the dashboard component silently re-fetches once ~5 s later). Only a never-cached (cold) account/week refreshes synchronously. Two latent concurrency bugs surfaced by the parallel refresh were fixed: DB writes are now one bounded `$transaction` per account behind a 5-slot semaphore (`withDbSlot`) so a fleet refresh can't exhaust Supabase's 15-client pool, and `RedditService.getAccessToken` is **single-flight** so a cold thundering-herd doesn't trigger concurrent (rejected) token mints. **Verified live** (10 real accounts): cold this-week **2.9 s** → warm **0.15–0.22 s** (no Reddit calls in log), stale load returns immediately with `refreshing:true` and the background refresh advances `refreshedAt`, last-week cold populates cleanly (one-time ~71 s completed-week crawl) → warm 0.7 s, error isolation holds under Reddit 429s (200 with isolated error rows), invalid range→400 / no-auth→401. Backend `nest build`+eslint and frontend `ng build` all clean; migration applied to Supabase; Prisma client regenerated.
- **Next up:** Remaining deferred features, each its own `/plan-feature`: **post-history views** (surface `/submitted` items in tables/history, reusing `getWeeklyPostCount`'s paging), **Payroll** reports, and **History caching** (Activity History still does its own O(accounts) live Reddit pass — could reuse the same metric-cache pattern). Optional polish: a scheduled poller to warm the cache proactively (currently lazy on view); precise Monday karma baselines via a weekly cron; shrinking the one-time cold last-week crawl.

## Key Decisions Made

- **Stack:** Angular 18+ / Angular Material · NestJS · Prisma · PostgreSQL — mirrors budgetwise for skill/knowledge reuse.
- **Auth:** JWT email/password for Dash accounts. Reddit OAuth is only for linking individual `RedditAccount`s, not for logging into Dash.
- **No wiki, no tickets.** Prompt-driven workflow with `PROJECT-STATUS.md` as the only persistent memory.
- **Reddit client:** plain `axios` against `oauth.reddit.com`. `snoowrap` is un-maintained; only reach for it if it saves real work.
- **Reddit auth = one shared Devvit token, not per-account OAuth.** A registered Reddit "web app" (client_id/secret) is no longer obtainable, so Dash reuses the token minted by `devvit login` (`~/.devvit/token`) — a standard Reddit OAuth2 refresh token, `scope=*`, from Reddit's first-party **public** Devvit client (no secret). Since comment history is public, one such token reads every tracked account by username, so accounts are tracked by username only (no per-account link/consent/token). Refresh must use the same Devvit client that minted the token: copy-paste login → `TWTsqXa53CexlrYGBWaesQ` (the code default), localhost login → `Bep8X2RRjuoyuxkKsKxFuQ` (set `REDDIT_OAUTH_CLIENT_ID`). Accepted trade-off: this is outside the token's intended use (plausible ToS risk) and Reddit can rotate the client id — hence it's config-overridable.
- **Prisma pinned to 6.x.** npm's latest is Prisma 7, which drops `url` from the datasource block and requires a `prisma.config.ts` + driver adapter passed to `PrismaClient`. For a scaffold that's unnecessary ceremony, so we pinned `prisma`/`@prisma/client` to `^6` (6.19.3) — classic `url = env("DATABASE_URL")` schema and adapter-free `PrismaService`. Revisit if a v7 feature is needed.
- **Angular Material 20 theme configured by hand.** `ng add @angular/material` installed the packages but its theming schematic errored (`azure/blue` "primary" bug) before writing config, and it did not add `@angular/animations`. We wired the theme in `styles.scss` (`mat.theme(...)`), animations via `provideAnimationsAsync()`, and installed `@angular/animations` manually.
- **Frontend versions:** Angular 20 / Material 20 (newer than the "18+" floor in CLAUDE.md) — that's what the CLI installed; standalone-components + signals patterns apply.
- **Auth = access + refresh tokens, admin-managed accounts.** No public registration: the seed script creates the first admin; admins create shillers via `POST /api/users` (role forced to `shiller` server-side). Access token 15m, refresh 7d. Refresh tokens are rotated on use and stored only as SHA-256 hashes in `RefreshToken`, so the DB can revoke/expire them without ever holding the raw token. Logout revokes only the presented (current-session) token. `bcrypt` at 12 rounds via `common/crypto/password.util.ts`.
- **Frontend session in `localStorage`.** Tokens + user cached under `dash.*` keys; the interceptor refreshes once on a 401 and force-logs-out to `/login` if that fails. Chosen for simplicity over httpOnly cookies for this internal tool.
- **DB = Supabase Postgres via the session pooler (port 5432).** That URL works for `prisma migrate`. Migrations applied directly from the dev machine.
- **UI follows the "Signal." prototype, not Material chrome.** `Dashboard_UI_Prototype.html` is the visual spec. The app shell is hand-built (custom dark sidebar/topbar), and shared visual primitives (cards, tables, pills) are global classes in `styles.scss`; Angular Material is kept only where it earns its keep (dialogs, form fields, menus, snackbars, the date-range picker). Charts are **inline SVG** — no ng2-charts/Chart.js dependency (it was listed in status but never actually installed).
- **Role → view mapping.** Dash has only `admin` + `shiller`. `admin` gets the prototype's manager/marketing overview (all accounts across shillers + Team & Access); `shiller` gets the employee overview (own accounts only). Enforced server-side in `getDashboard` (scope by role) and by `roleGuard('admin')` on `/team`.
- **One dashboard aggregation endpoint, now served from a stale-while-revalidate cache.** `GET /api/reddit/dashboard` computes per-account weekly counts + karma server-side so the browser never fans out N Reddit calls. It no longer polls Reddit in the request path: reads come from the `AccountWeekMetric` cache + `RedditAccount` snapshot (instant); Reddit is re-polled only in the background when a metric is >5 min stale, or synchronously the first (cold) time an account/week is seen. Refresh work is bounded — parallel Reddit reads (throttle-gated) but per-account writes are one atomic `$transaction` behind a 5-slot semaphore, and token minting is single-flight — after live testing showed unbounded `Promise.all` exhausting the Supabase pooler and racing concurrent token mints. Chosen over in-memory caching (survives restarts) and over a scheduled poller (refresh stays lazy on view).
- **Weeks are Monday-based.** `startOfWeek` = Monday 00:00 local; `this-week` filters Mon→now, `last-week` the prior Mon–Sun. The topbar label always shows the full Mon–Sun span (e.g. "Jul 6 – Jul 12, 2026").
- **Karma trend = write-through weekly snapshots, no cron; baseline seeded at link, trend shows from week 1.** Karma is otherwise read live and never stored. A `KarmaSnapshot` baseline is captured **at account-link time** (`addAccount` → `captureLinkBaseline`, reusing the karma from the validation `/about` — no extra Reddit call) and, as a safety net for already-linked accounts, on each dashboard load `buildDashboardRow` does a **create-only** upsert for the current `weekStart`. The first karma seen that week becomes an immutable baseline; later loads the same week don't change it. `karmaThisWeek` = live − this-week baseline; `karmaLastWeek` = this-week baseline − last-week baseline (null until both exist). The UI gates the chart + Δ column on `karmaThisWeek` (not `karmaLastWeek`), so this week's gain shows immediately (single this-week bar); the paired "vs last week" bar + green/orange coloring appear once a second weekly baseline exists — we don't fabricate a prior week. Chosen over a `@nestjs/schedule` cron because the backend isn't guaranteed to run at week boundaries. Costs one upsert + one batched select per load — **no extra Reddit calls**.
- **Weekly quotas are per-account, configured per shiller.** `User.weeklyCommentQuota`/`weeklyPostQuota` (defaults 50/5) are the target *each of that shiller's accounts* should hit in a week — matching CLAUDE.md's "which accounts already met this week's quota." The dashboard's quota-progress cards aggregate the target as Σ over in-scope accounts of that account's owner's quota (shiller = quota × their account count; admin = summed across all accounts). Progress bars cap at 100% while the count label stays uncapped; a 0 target (no accounts / quota 0) shows the count with no bar. Chosen over a separate quota table (simpler, edited inline in the existing Edit-shiller dialog) and over a single team-wide target (per-account scales naturally with account volume). This added a second Reddit read per account per dashboard load (`/submitted`), so per-load cost is now ~2–3 calls/account — a reason caching moves up the priority list.
- **Admins (and therefore yourself) can't be deleted.** `UsersService.remove` rejects deleting any `admin`; since every `/api/users` caller is an admin, that single rule also blocks self-deletion. Role is not editable via `PATCH` — created users stay `shiller` (only the seed makes admins).

## Known Issues

- **`REDDIT_REFRESH_TOKEN` must be set in `backend/.env`** (agent-write-blocked — user adds it). Extract it once from `~/.devvit/token` with:
  `node -e "const fs=require('fs');const o=JSON.parse(fs.readFileSync(process.env.USERPROFILE+'/.devvit/token'));console.log(JSON.parse(Buffer.from(o.token,'base64').toString()).refreshToken)"`
  Only `REDDIT_USER_AGENT` + `REDDIT_REFRESH_TOKEN` are required now (optionally `REDDIT_OAUTH_CLIENT_ID` if the token came from the localhost devvit login flow). `REDDIT_CLIENT_ID`/`REDDIT_CLIENT_SECRET`/`REDDIT_REDIRECT_URI` are **no longer read** and can be removed. The Reddit-API path itself is verified live; only the `.env` value + a browser click-through remain.
- **`backend/.env` and `.env.example` are agent-write-blocked.** The real `.env` was hand-created by the user with generated JWT/encryption secrets. `.env.example` still needs `JWT_ACCESS_SECRET`/`JWT_ACCESS_EXPIRES`, `JWT_REFRESH_SECRET`/`JWT_REFRESH_EXPIRES`, and `ADMIN_EMAIL`/`ADMIN_PASSWORD` added by hand if not already present.
- **Seeded admin uses a placeholder password.** `admin@dash.local` / `DashAdmin123!` — change it (re-seed after deleting the row, or add a password-change flow).
- **Frontend bundle is back under the 500 kB budget** (~398 kB initial after the redesign dropped the Material shell in favor of the custom dark shell). No budget warning currently.
- **Dashboard load is cached (now with a manual Refresh); History is not.** The dashboard serves from the `AccountWeekMetric` cache and revalidates in the background — no Reddit calls in the hot path — and a **Refresh button** (`?refresh=true`) forces a synchronous re-poll of every in-scope account. Note: that forced re-poll is **correct but slow** — ~41 s for 10 admin accounts through the 60/min throttle (it re-polls the whole fleet); the button spins and keeps current numbers visible, but a future optimization could scope the force to just the viewing shiller's accounts or refresh incrementally. **Activity History still does an O(accounts) live Reddit pass** (re-derives the account list from `dashboard()` then fetches comments) — reuse the same metric-cache pattern (or a shared cached comments store) before it scales. Also: the **cold** dashboard path still pays live Reddit cost once per (account, week) — this-week is ~3 s for ~10 accounts, but a **completed last-week** cold load can take ~60–70 s (up to ~10 comment + ~10 post pages/account through the 60/min throttle); an optional pre-warm/poller would hide that first-view cost.
- **The comment/post crawls deliberately omit `sort=new`.** `getComments`/`getWeeklyPostCount` stop at the first item older than the week start, which relies on the listing being newest-first — and Reddit's **default** user-listing order already is. Passing `sort=new` explicitly makes Reddit serve a stale sorted-listing edge cache that freezes counts at old values, so it is intentionally left off (documented inline in `reddit.service.ts`). If a future change ever adds a different sort mode, keep the early-stop crawl chronological or it will under-count.
- **Redesign not exercised in a browser this session.** Verified at `ng build` (AOT) + live-API level. The shiller-scoped dashboard/accounts/history views are covered by the service where-clause and route guards but weren't clicked through as a shiller (no known shiller password — the seed creates only the admin). Worth a manual pass as both roles at 375px and 1440px.
- **`account-comments` / `shiller-detail` views are dark-themed but not fully reskinned** to the prototype's card/table language — they inherit the Material dark theme and remain functional. Reskin if a fully consistent look is wanted.
- **Prisma prints a `package.json#prisma` seed-config deprecation warning** (removed in Prisma 7). Harmless on the pinned v6; revisit if/when upgrading to a `prisma.config.ts`.
- **Frontend login UI not exercised in a browser this session** — verification was API-level (12/12) plus a clean `ng build`. Worth a manual click-through of `/login` → dashboard → logout.
